# 游戏开发全流程架构

> 以网页开发经验为锚点，对照理解游戏开发的核心差异。
> 本文档面向"懂网页前后端，想入门游戏开发"的读者。

---

## 一、核心思维转变

### 网页模式（你熟悉的）

```
用户操作 → HTTP请求 → 后端处理 → 返回数据 → 重新渲染（DOM diff）
          ↑ 事件驱动，请求-响应                                    ↓
          └────────────── 等待用户下次操作 ←───────────────────────┘
```

**特点：**
- 事件触发，无事发生时页面"休眠"
- 状态通过 API 拉取/提交
- 渲染由浏览器 DOM 引擎自动处理 diff

### 游戏模式

```
每 1/60 秒执行一帧:
  ┌─────────────────────────────────────────────────────┐
  │  处理输入 → 更新游戏逻辑 → 物理模拟 → 碰撞检测 → 渲染  │
  └─────────────────────────────────────────────────────┘
           ↑ 永不停止的循环，无论是否有操作 ↓
```

**特点：**
- 持续不断的帧循环，没有"休眠"状态
- 状态是增量计算（上一帧 + 变化量 = 当前帧）
- 手动管理渲染、资源、内存

## 二、游戏架构八大分层

### 1. 渲染层（对应网页的 DOM/CSS）

| 网页 | 游戏 |
|------|------|
| HTML+CSS 声明式布局 | Canvas/WebGL/OpenGL 命令式绘制 |
| 浏览器处理重排重绘 | 每帧手动清除 → 排序 → 绘制 → 提交 |
| 事件冒泡自动处理 | 需手动实现点击检测（射线检测/色彩检测） |

**选型：**
- 网页 2D 游戏：Canvas 2D API / PixiJS（WebGL 封装的 2D 引擎）
- 3D 游戏：Three.js（Web）/ Unity / Unreal
- 独立游戏：Godot（开源全功能引擎）

### 2. 场景/关卡管理（对应前端路由）

```
网页路由:  /home  → 加载 HomePage → 请求数据 → 渲染
游戏场景: MenuScene → 加载资源 → 初始化实体 → start() → update() 循环
          战斗场景 → 卸载Menu资源 → 加载战斗资源 → ...
```

场景管理器职责：
- 资源生命周期（进入加载，离开卸载）
- 场景切换过渡（淡入淡出、Loading 进度）
- 场景间数据传递（如：选关界面 → 关卡场景 → 结算界面）

### 3. 游戏对象系统（对应前端组件树）

**方案 A：继承模式（OOP 传统）**

```
GameObject (基类)
  ├── Player (extends GameObject) — 玩家角色
  ├── Enemy (extends GameObject) — 敌人
  ├── Bullet (extends GameObject) — 子弹
  └── Item (extends GameObject) — 道具
```

**方案 B：ECS 模式（Entity-Component-System，现代游戏主流）**

```
实体 = 空容器，只有 ID
  ├── 玩家实体 = [Position, Velocity, Sprite, Health, PlayerInput]
  ├── 敌人实体 = [Position, Velocity, Sprite, Health, AI]
  └── 子弹实体 = [Position, Velocity, Sprite, Damage]

系统（遍历匹配的实体组合）:
  移动系统 → 遍历 [Position + Velocity] 的实体 → 更新位置
  渲染系统 → 遍历 [Position + Sprite] 的实体 → 绘制
  AI系统   → 遍历 [AI + Position] 的实体 → 决策
```

ECS 优势：高度解耦、数据驱动、缓存友好、便于热更新

### 4. 帧循环与状态管理

```
function gameLoop(timestamp) {
    const dt = timestamp - lastTimestamp;   // 帧间隔时间(ms)
    lastTimestamp = timestamp;
    
    // 固定时间步长（物理稳定性）
    accumulator += dt;
    while (accumulator >= FIXED_DT) {
        fixedUpdate(FIXED_DT);             // 固定步长更新
        accumulator -= FIXED_DT;
    }
    
    render(accumulator / FIXED_DT);         // 渲染（带插值因子）
    requestAnimationFrame(gameLoop);
}
```

**状态管理的区别：**
- 网页：不可变状态（immutable），`setState` 触发重渲染
- 游戏：可变状态（mutable），每帧增量更新

### 5. 物理引擎（网页没有的层）

自动处理：碰撞检测、刚体运动、重力、摩擦力、关节约束

```
每帧物理流程:
  1. 应用力（重力、力场）
  2. 更新速度（F = ma，v += a × dt）
  3. 更新位置（x += v × dt）
  4. 碰撞检测 → 触发回调（onCollisionEnter/Stay/Exit）
  5. 解算约束（关节、弹簧）
  6. 修正穿透位置
```

**选型：**
- 2D：Box2D（C++ 移植版很多）、Matter.js（JS）
- 3D：Cannon.js、Ammo.js、内置引擎物理

### 6. 动画系统

| 类型 | 说明 | 适用 |
|------|------|------|
| 逐帧动画 | 快速切换 sprite 图 | 2D 角色行走/攻击 |
| 骨骼动画 | 旋转/位移骨骼层级 | 3D 角色 |
| 补间/插值 | 对属性做平滑过渡 | UI 动画、摄像机移动 |
| 动画状态机 | 管理状态切换 + 过渡 + blend | 复杂角色（idle→walk→run→jump→attack→die） |

### 7. 网络层（HTTP → UDP/WebSocket）

```
网页网络:
  GET /api/move → 等待 → 200 {"x":100,"y":200} → 更新
  延迟可接受（200ms-500ms）

游戏网络（实时多人）:
  客户端每帧发送: [输入序列号, 按键状态]
  服务器每 tick 发送: [世界状态帧, 权威位置]
  延迟要求: < 100ms，否则玩家感觉卡顿

关键技术:
  客户端预测(Client Prediction)
    不等服务器确认，立刻执行本地输入，消除"输入→响应"延迟
  
  服务器回滚(Server Reconciliation)
    当服务器下发的权威状态与本地预测不一致时，回滚修正
  
  插值(Interpolation)
    在两个已知服务器状态之间平滑渲染其他实体
  
  延迟补偿(Lag Compensation)
    服务器"回到过去"检测命中（解决"我看到他在 A，其实他在 B"）
```

**选型建议：**
- 实时同步（FPS/格斗）：WebSocket + 自定义状态同步
- 回合制/卡牌：WebSocket 或 HTTP + 轮询
- 非实时（单机/异步）：不需要网络层

### 8. 资源管理（对应 Webpack/Vite）

```
网页: JS+CSS 打包 → CDN → 按路由懒加载

游戏资产管线:
  原始资产(PSD/FBX/WAV) → 引擎导入 → 压缩 → 打包 → 运行时加载

常见格式:
  模型: .glb/.glTF（推荐） / .fbx / .obj
  贴图: .png（有损压缩）/ .ktx2（GPU 压缩）
  音频: .mp3 / .ogg / .wav

加载策略:
  同步加载: 场景切换时加载全部资源（简单但会卡）
  异步预加载: 当前场景运行时，后台加载下一场景
  流式加载: 开放世界，只加载玩家周围区块
  对象池: 子弹/敌人等频繁创建销毁的对象，复用而不是新建
```

## 三、完整帧循环时序（@60fps 每帧 16.7ms）

```
┌──────────────┬──────────────────────────────────────┐
│   阶段        │  耗时估算    │  说明                   │
├──────────────┼──────────────────────────────────────┤
│ 1. 输入采集   │  ~0.5ms     │  键盘/鼠标/手柄状态       │
│ 2. 游戏逻辑   │  ~2ms       │  AI决策、事件触发、状态变更 │
│ 3. 物理模拟   │  ~3ms       │  刚体运动、碰撞检测        │
│ 4. 动画更新   │  ~1ms       │  骨骼/Tween 插值          │
│ 5. 音频混合   │  ~0.5ms     │  播放/淡入淡出/3D音效      │
│ 6. 渲染提交   │  ~5ms       │  清屏 → 排序 → 绘制 → 交换 │
│ 7. 网络收发   │  ~1ms       │  WebSocket 发包/收包       │
├──────────────┼──────────────────────────────────────┤
│ 合计         │  ~13ms      │  剩余 ~3.7ms 给 GC/其他    │
└──────────────┴──────────────────────────────────────┘
```

## 四、网页游戏技术栈推荐

### 轻量 2D 游戏

| 层 | 推荐 | 说明 |
|----|------|------|
| 渲染 | Canvas 2D API / PixiJS | PixiJS 提供 WebGL 加速的 2D 渲染 |
| 物理 | Matter.js | 2D 刚体物理引擎，API 友好 |
| 游戏循环 | requestAnimationFrame | 浏览器原生 API |
| 动画 | GSAP | 工业级补间动画库 |
| 音频 | Howler.js | 跨浏览器音频播放 |
| 网络 | WebSocket | 原生支持，二进制帧高效 |
| 打包 | Vite + TypeScript | 开发体验好，HMR 快 |
| ECS | bitecs | 极简轻量 ECS 库 |

### 重型/3D 游戏

- **Three.js** + **Cannon.js** / **Ammo.js**（Web 3D）
- **Unity**（C#，跨平台，资产丰富）
- **Godot**（开源，GDScript/C#，轻量）
- **Cocos Creator**（国产，TypeScript，2D 强项）

## 五、数据模型对比

```
网页数据模型（关系型, CRUD）:
  User { id, name, level, gold }
  Item { id, owner_id, type, count }
  → SELECT * FROM items WHERE owner_id = ?

游戏数据模型（组件化, 空间索引）:
  运行时:
    Entity:
      - Transform { x, y, z, rotation, scale }
      - Stats { hp, maxHp, speed, attack, defense }
      - Inventory { slots: [{itemId, count}] }
      - AIState { mode: "patrol"|"chase"|"attack", targetId }
    SpatialHash / QuadTree: 空间分区加速碰撞检测

  持久化:
    Redis（热数据: 在线玩家位置、血量）
    MongoDB（冷热混合: 玩家数据、背包）
    Protocol Buffers（二进制序列化，高效传输）
```

## 六、核心术语速查

| 术语 | 说明 |
|------|------|
| **帧(Frame)** | 一次完整的更新+渲染循环 |
| **Delta Time (dt)** | 两帧之间的时间间隔，单位 ms 或 s |
| **Tick** | 服务器/物理的固定时间步更新 |
| **Interpolation** | 在两个状态之间平滑插值 |
| **Lerp** | 线性插值，位置/颜色过渡的数学基础 |
| **Sprite** | 2D 图像/精灵，游戏中最基本的视觉元素 |
| **Tilemap** | 瓦片地图，用格子拼出地图 |
| **Bounding Box** | 包围盒，用于碰撞检测的简化形状 |
| **Raycast** | 射线检测，从一点发出射线检测碰撞 |
| **Object Pool** | 对象池，复用对象避免内存抖动 |
| **Scene Graph** | 场景图，树形结构的场景对象层级 |
| **RigidBody** | 刚体，物理引擎中受力的物体 |
| **Blend Tree** | 动画混合树，平滑过渡不同动画 |
| **Spatial Hash** | 空间哈希，将二维/三维空间分块加速查询 |

## 七、学习路径参考

1. **先做一个小游戏走通全流程**
   - 用 Canvas 2D + requestAnimationFrame
   - 实现：玩家移动 → 碰撞检测 → 敌人生成 → 得分/血量
   - 这就是完整的"游戏"了，后面都是这个基础的扩展

2. **引入引擎（PixiJS 或 Godot）**
   - 解决手写 Canvas 的性能和功能瓶颈
   - 学习场景管理、Sprite 图集、粒子系统

3. **深入某一块**
   - 物理：研究 Box2D 工作原理
   - 网络：实现一个简单的帧同步 demo
   - 性能：分析 Draw Call、内存管理、GC 优化

4. **参考资源**
   - [Game Programming Patterns](https://gameprogrammingpatterns.com/)（经典免费书）
   - MDN 的 Canvas 教程
   - PixiJS 官方示例
