## 1. 项目概览对比

### 1.1 基本信息对比

| 维度 | 梨园生死 | 斯坦福小镇 |
|------|----------|----------|
| **项目性质** | 黑客松作品（AI叙事游戏） | 学术论文配套代码（AI智能体研究） |
| **核心目标** | 玩家驱动叙事，AI生成对话和剧情 | 研究AI智能体在虚拟社区的自主行为 |
| **NPC数量** | 5个故事NPC + 若干城镇NPC | 25个智能体（可扩展到更多） |
| **地图规模** | 80×50瓦片（小镇+6个子场景） | 140×100瓦片（完整小镇） |
| **坐标系统** | 瓦片坐标(col,row) → 像素坐标(x,y) | 瓦片坐标(x,y) → 像素坐标(x,y) |
| **技术栈** | Phaser 3 + Vite + FastAPI + SQLite | Django + Phaser.js + Python仿真引擎 |
| **LLM调用** | 对话时实时调用（SSE流式） | 每步仿真调用（串行） |
| **数据存储** | SQLite + JSON存档 | JSON文件（environment/*.json, movement/*.json） |

### 1.2 核心设计理念对比

**梨园生死**：
- **玩家中心**: 玩家是叙事的核心驱动力，NPC围绕玩家互动
- **章节式叙事**: 六章固定结构，AI生成具体对话和任务
- **关系值驱动**: NPC对玩家的态度由数值(-100~100)决定
- **任务投票机制**: NPC通过投票决定是否完成任务

**斯坦福小镇**：
- **智能体中心**: 每个智能体有独立的生活和目标
- **开放式模拟**: 无预设剧情，行为完全由AI驱动
- **记忆驱动**: 智能体行为由记忆检索和反思驱动
- **涌现式叙事**: 故事从智能体互动中自然涌现

## 2. 核心架构差异分析

### 2.1 NPC智能程度对比

#### 梨园生死的NPC架构

```
NPC对话请求
    ↓
PromptBuilder.build_dialogue_messages()
    ↓ 注入：人设 + 对话历史 + 章节上下文 + 关系值
LLM调用
    ↓
ResponseParser.parse_dialogue_response()
    ↓ 提取：dialogue_text + relationship_delta + options + task_progress
应用结果：更新关系值、记录对话、处理任务投票
```

**特点**：
- NPC是"对话响应器"，只在玩家交互时激活
- 记忆只有对话历史（压缩摘要）
- 无自主规划能力

#### 斯坦福小镇的NPC架构

```
每步仿真（游戏内10秒）
    ↓
Perceive（感知）: 观察周围环境，记录到记忆流
    ↓
Retrieve（检索）: 根据当前情境检索相关记忆
    ↓
Reflect（反思）: 定期生成insights（重要性归零时触发）
    ↓
Plan（规划）: 生成日程表，分解到5分钟粒度的动作
    ↓
Execute（执行）: 按规划执行动作，更新位置状态
    ↓
Converse（对话）: 与其他智能体对话，更新双方记忆
```

**特点**：
- NPC是"自主智能体"，有完整认知闭环
- 记忆系统：关联记忆（Recency+Relevance+Importance）+ 空间记忆（MemoryTree）+ 记忆流（Chronological）
- 有长期规划能力（日报→行动）

### 2.2 记忆系统对比

#### 梨园生死的记忆系统

| 记忆类型 | 实现方式 | 容量限制 | 检索方式 |
|---------|---------|---------|---------|
| 对话历史 | `dialogue_history: list[DialogueTurn]` | 无限制（但会压缩） | 全部注入Prompt |
| 压缩摘要 | `compressed_summaries: dict[str, str]` | NPC级别摘要 | 注入Prompt |
| 事件记录 | `events_triggered: set[str]` | 无限制 | 注入Prompt |
| 物品知识 | `npc_knowledge: dict[str, str]` | 物品级别 | 注入Prompt |

**问题**：
1. 对话历史超线性增长 → Token消耗大
2. 无语义检索 → 无关信息也会注入
3. 无空间记忆 → NPC不知道自己在哪里、周围有什么

#### 斯坦福小镇的记忆系统

| 记忆类型 | 实现方式 | 容量限制 | 检索方式 |
|---------|---------|---------|---------|
| 关联记忆 | `associative_memory.json` | 无限制 | Recency(1/d) + Relevance(LLM评分) + Importance(人工标注) 加权 |
| 空间记忆 | `spatial_memory.json`（MemoryTree） | World→Sector→Arena→GameObject | 树形遍历 |
| 记忆流 | `memory_stream.json` | 无限制 | 时间排序 |

**优势**：
1. 语义检索 → 只注入相关记忆
2. 空间记忆 → NPC知道自己在哪、怎么去其他地方
3. 记忆流 → 完整的时间线

### 2.3 世界建模对比

#### 梨园生死的世界建模

```
地图：80×50瓦片
    ↓
碰撞检测：frontend/src/scenes/modules/CollisionEditor.js（编辑工具）
    ↓
子场景：6个独立地图（戏台、茶馆、码头、祠堂、父亲故居、墓地）
    ↓
NPC位置：config.py中静态定义，无动态更新
```

**问题**：
1. NPC位置静态 → 无移动能力
2. 无空间语义 → "茶馆"只是一个场景名，NPC不知道茶馆里有什么
3. 无物体交互 → NPC不能操作物品

#### 斯坦福小镇的世界建模

```
地图：140×100瓦片
    ↓
碰撞矩阵：maze.py（self.tiles[y][x]）
    ↓
空间语义：MemoryTree（World→Sector→Arena→GameObject）
    ↓
物体交互： Persona类有current_action字段，可记录"在X处做Y"
```

**优势**：
1. 动态位置 → NPC可以自由移动
2. 空间语义 → NPC知道"Hobbs Cafe"在哪个扇区、有哪些物体
3. 物体交互 → NPC可以"坐在沙发上"、"拿起杯子"

## 3. 斯坦福小镇的优势赋能点

### 3.1 直接可赋能的点

#### 3.1.1 增强NPC自主性

**现状**（梨园生死）：
- NPC只在玩家交互时"醒来"
- 无自主行为 → 玩家不找NPC，NPC就什么都不做

**赋能后**：
- NPC有自己的日程表（基于人设和当前章节）
- 玩家不在时，NPC也在"生活"（排练、喝茶、聊天）
- 玩家回到茶馆，可能看到陈师傅在教小华唱戏 → **世界真实感↑**

**实现方式**：
1. 引入斯坦福小镇的`Plan（规划）`模块
2. 为每个NPC生成日报（基于人设 + 当前章节目标）
3. 在后台仿真循环中执行规划

#### 3.1.2 增强记忆系统

**现状**（梨园生死）：
- 对话历史全部注入 → Token消耗大，无关信息多
- 无空间记忆 → NPC不知道自己在哪

**赋能后**：
- 引入关联记忆检索 → 只注入相关记忆
- 引入空间记忆 → NPC知道"我在茶馆，梅姨在柜台后面"
- 引入记忆流 → 完整的时间线，支持"你还记得X天前..."

**实现方式**：
1. 引入斯坦福小镇的`Retrieve（检索）`模块
2. 用Recency+Relevance+Importance加权检索
3. 引入MemoryTree（简化版，适配80×50地图）

#### 3.1.3 增强世界真实感

**现状**（梨园生死）：
- NPC位置静态 → 陈师傅永远在茶馆(38,14)
- 无NPC社交 → 陈师傅和小华永远不会对话

**赋能后**：
- NPC会移动 → 陈师傅可能从茶馆走到戏台
- NPC会社交 → 玩家不在时，陈师傅和小华在戏台排练
- 玩家"偷听" → 可以听到NPC之间的对话

**实现方式**：
1. 引入斯坦福小镇的`Execute（执行）`模块
2. 用波前扩散算法做寻路
3. 引入`Converse（对话）`模块（NPC-NPC对话）

### 3.2 需要改造的点

#### 3.2.1 从"仿真步"到"游戏帧"

**斯坦福小镇**：
- 仿真在主循环`reverie.py`中运行
- 每步=游戏内10秒，每秒钟推进多步
- 完全后台运行，前端轮询

**梨园生死**：
- 无后台仿真循环
- 所有行为由玩家交互触发
- 前端是游戏主循环（Phaser.js的`update()`）

**改造方案**：
1. **方案A（推荐）**：在后端添加"后台仿真循环"
   - 用`asyncio`任务，每隔N秒推进一个仿真步
   - 前端通过WebSocket/SSE接收NPC位置更新
   
2. **方案B（轻量）**：在前端添加"NPC行为系统"
   - 在Phaser的`update()`中更新NPC位置
   - 后端只提供"NPC日程表"

#### 3.2.2 从"25个智能体"到"5个NPC"

**斯坦福小镇**：
- 25个智能体，每步串行调用LLM → 慢
- 用`multiprocessing`并行化（但仍有延迟）

**梨园生死**：
- 5个NPC，可以承受更频繁的LLM调用
- 但也要考虑成本

**改造方案**：
1. 为5个NPC建立"行为缓存"
   - 规划结果缓存1小时（游戏内时间）
   - 对话结果缓存5分钟
2. 用"重要性过滤"减少LLM调用
   - 只有重要事件才调用LLM
   - 日常对话用模板库

### 4.2 分阶段实施计划

#### 阶段一：记忆系统升级（2周）

**目标**：引入关联记忆检索，解决Token消耗问题

**任务**：
1. 实现`AssociativeMemory`类（基于斯坦福小镇）
   - 文件：`backend/memory/associative_memory.py`
   - 接口：`add_memory(text, recency, relevance, importance)`
   - 检索：`retrieve_relevant_memories(current_context) -> list[str]`

2. 改造`PromptBuilder`
   - 原来：注入全部对话历史
   - 改造后：用`retrieve_relevant_memories()`检索相关记忆，只注入前K条

3. 数据迁移
   - 将现有`dialogue_history`迁移到`associative_memory`
   - 为每条历史标注recency、relevance、importance

**预期效果**：
- Token消耗降低50%~70%
- NPC对话更"有记忆"，不会忘记前几天的事

#### 阶段二：NPC自主性升级（3周）

**目标**：NPC有自己的日程表，会在玩家不在时"生活"

**任务**：
1. 实现`Plan（规划）`模块
   - 文件：`backend/agents/planner.py`
   - 输入：NPC人设 + 当前章节 + 记忆
   - 输出：日报（JSON格式，包含时间戳和动作）

2. 实现`Execute（执行）`模块
   - 文件：`backend/agents/executor.py`
   - 输入：日报 + 当前时间
   - 输出：NPC当前位置 + 当前动作

3. 添加后台仿真循环
   - 用`asyncio.create_task()`创建后台任务
   - 每隔60秒（游戏内时间）推进一个仿真步
   - 通过WebSocket向前端推送NPC位置更新

**预期效果**：
- NPC会移动（在地图上看到陈师傅从茶馆走到戏台）
- NPC会"生活"（玩家不在时，NPC也在做事）

#### 阶段三：世界建模升级（2周）

**目标**：引入空间记忆和物体交互

**任务**：
1. 实现简化版`MemoryTree`
   - 文件：`backend/memory/spatial_memory.py`
   - 结构：World → Sector（8×8瓦片） → Arena（建筑） → GameObject（物品）
   - 接口：`observe_tile(x, y, description)` + `find_path(target_x, target_y)`

2. 改造地图数据
   - 在`data/scripts/liyuan_shengsi/`下添加`spatial_definitions.yaml`
   - 定义所有Sector、Arena、GameObject

3. 实现物体交互
   - 在`NPCState`中添加`current_action: str`字段
   - NPC可以"坐在茶馆的椅子上"、"拿起京胡"

**预期效果**：
- NPC知道自己在哪、周围有什么
- NPC可以操作物品（增强沉浸感）

#### 阶段四：NPC社交系统（2周）

**目标**：NPC之间会对话，形成"小镇谣言"

**任务**：
1. 实现`Converse（对话）`模块
   - 文件：`backend/agents/conversation_manager.py`
   - 触发条件：两个NPC在同一位置 + 都在"闲逛"状态
   - 输出：对话记录（保存到双方记忆）

2. 实现"谣言传播"机制
   - 玩家与NPC对话时，NPC可能说"我听陈师傅说..."
   - 形成"小镇消息网"

**预期效果**：
- 世界更真实（NPC不是孤岛）
- 玩家可以通过不同NPC了解"小镇动态"

## 6. 风险与挑战

### 6.1 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM调用成本增加 | NPC自主性升级后，LLM调用频率↑ | 用"重要性过滤" + "缓存机制" |
| 性能下降 | 后台仿真循环占用资源 | 用`asyncio` + 限制仿真频率 |
| 记忆系统复杂度↑ | 新系统bug可能导致NPC"失忆" | 保留旧系统作为fallback |

### 6.2 设计风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| NPC太自主，玩家失控 | 玩家觉得"剧情不受控" | 用"章节目标"约束NPC行为 |
| 叙事一致性破坏 | NPC自主行为导致剧情矛盾 | 用"叙事导演AI"把控节奏 |
| 复杂度过高，难以调试 | 出现问题难以定位 | 完善的日志系统 + 可视化调试工具 |
