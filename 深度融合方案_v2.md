## 1. 两个项目的基因对比

### 1.1 不是"好与坏"，是"游戏类型差异"

梨园生死和斯坦福小镇代表了两种根本不同的游戏哲学：

| 维度 | 梨园生死 | 斯坦福小镇 | 互补性 |
|------|----------|----------|--------|
| **叙事结构** | 6章线性章节，预定义剧情弧 | 无结构，完全涌现式 | 梨园提供叙事骨架 |
| **NPC 行为** | 被动响应，玩家驱动 | 自主规划，自我驱动 | 小镇提供行为引擎 |
| **时间感知** | 章节驱动，无实时概念 | 仿真步进（10秒/步） | 小镇提供时空框架 |
| **记忆系统** | 对话历史（线性存储） | 关联记忆+空间记忆+记忆流 | 小镇提供记忆检索 |
| **世界模型** | 6个静态子场景 | 140×100瓦片动态世界 | 小镇提供空间语义 |
| **玩家角色** | 故事主角，核心驱动力 | 不存在，纯粹观察者 | 梨园提供玩家定位 |
| **NPC 关系** | 数值化好感度(-100~100) | 记忆密度驱动（非数值） | 互补可融合 |

### 1.2 梨园生死的「已经很强」的部分

以下系统是梨园生死已有的核心竞争力，融合时**不替换、只增强**：

1. **章节式叙事引擎** — 6章+序章，LLM任务规划+NPC共识投票 → 这是"可控叙事"的关键
2. **子场景系统** — 6个室内场景，进入/离开编排，状态保存恢复 → 比斯坦福小镇的单一地图更丰富
3. **物品叙事系统** — 10+件叙事物品，NPC特殊认知，发现旁白 → 斯坦福小镇完全没有
4. **安全体系** — 4层安全（Agent行为/身份/数据/异常）→ 生产级标准
5. **存档系统** — 6槽位后端持久化 → 斯坦福小镇没有存档概念
6. **BGM/色调/立绘** — 完整的美术音乐管线 → 比学术原型强太多

### 1.3 梨园生死的「最需要增强」的部分

以下是对标斯坦福小镇后的关键差距：

1. **NPC 自主性 = 零** — NPC 永远站在原地，永远等玩家来对话，永远被动
2. **NPC-NPC 互动 = 零** — 陈师傅和小华同在戏台，但从不会说话
3. **空间感知 = 零** — NPC 不知道"我在茶馆"意味着什么，不知道周围有什么
4. **长期记忆 = 零** — 对话历史是线性列表，无检索，无反思
5. **世界变化 = 零** — 小镇永远是同一幅画面，没有任何"时间流逝"的感觉

## 2. 设计支柱：融合后的游戏体验

### 2.1 「梨园生死 2.0」的设计支柱

**① 记忆即真相**
> 你不是在"刷好感度"，你在帮NPC拼凑记忆碎片。陈师傅对小华的师徒情、老周对戏班的愧疚、梅姨对父亲的爱慕——这些情感在对话中不是"数值变化"，而是"记忆片段逐渐浮现"。

**② 小镇会呼吸**
> 当你不在戏台时，陈师傅在拉琴，小华在练功，老周在打盹。当你走进茶馆时，梅姨正在泡一壶新到的龙井——她记得你三天前说过"改天来尝"。这不是你在操作NPC，是NPC在"生活"。

**③ 因果有回响**
> 你在第一章帮小华修好了漏雨的屋顶 → 第三章暴雨来袭时，小华主动来茶馆找你"快去看戏台！" → 这不是脚本触发，是记忆网络检索到的"你帮他修过屋顶 → 他觉得你会关心戏台 → 他来找你"。

**④ 章节是交响乐，不是关卡**
> 6章不是"打完第一章才能进第二章"的关卡，而是"6个主题乐章"。NPC的自主行为和你的行动共同决定每章的"演奏方式"：你可能在第二章就触发了第三章的核心事件（提前发现照片触发记忆觉醒），也可能第四章比第三章更短（因为你已经提前了解了足够多）。

## 4. 系统一：NPC 自主生活系统

### 4.1 设计目标

让5个故事NPC + 3个城镇NPC在玩家不在场时也"生活"。不是完全的斯坦福小镇式自由（那会破坏章节叙事），而是**在章节框架约束下的自主行为**。

### 4.2 核心机制：约束式日程规划

```
NPC 日报生成流程：

输入：
  ① NPC 人设卡（chen_shifu.yaml）
  ② 当前章节目标（chapters.yaml → success_condition）
  ③ 当前记忆流（最近48小时游戏内时间）
  ④ 章节约束规则（由叙事导演下发）

LLM 生成：
  {
    "schedule": [
      {"time": "06:00", "action": "起床", "location": "茶馆二楼"},
      {"time": "06:30", "action": "泡茶", "location": "茶馆一楼柜台"},
      {"time": "07:00", "action": "开门营业", "location": "茶馆门口"},
      {"time": "09:00", "action": "去戏台看看", "location": "戏台"},
      {"time": "12:00", "action": "回来午休", "location": "茶馆二楼"},
      ...
      {"time": "22:00", "action": "关门", "location": "茶馆"}
    ],
    "constrained_areas": ["茶馆", "戏台", "码头"],  // 章节约束：本章此NPC不出这些区域
    "idle_dialogue_topics": ["戏班往事", "父亲的手稿", "新到的龙井"],
    "important_actions": [
      {"time": "09:00", "action": "去戏台送茶", "reason": "陈师傅昨天说嗓子不舒服"}
    ]
  }
```

**与斯坦福小镇的差异**：
- 斯坦福小镇的规划是"完全自由"的（NPC可以决定去任何地方做任何事）
- 梨园生死的规划是"章节约束"的（NPC的行动范围由当前章节的叙事需求决定）
- 这确保了NPC不会在第三章"跑去码头再也不回来"而导致关键对话无法触发

### 4.3 NPC 移动系统

```javascript
// 后端仿真循环（asyncio 后台任务）
async def npc_autonomy_loop(session):
    while not session.game_ended:
        for npc_id, npc in session.npcs.items():
            # 1. 检查当前时间应该做什么
            current_action = get_schedule_action(npc, game_time)
            
            # 2. 如果需要移动，计算路径
            if current_action.location != npc.current_location:
                path = pathfind(npc.position, current_action.location)
                # 沿路径逐步移动，每步通知前端
                for step in path:
                    npc.position = step
                    await ws_broadcast(session, {
                        "type": "npc_move",
                        "npc_id": npc_id,
                        "position": step
                    })
                    await asyncio.sleep(MOVE_SPEED)  # 每步间隔
            
            # 3. 如果需要对话（NPC-NPC），触发
            if should_converse(npc, nearby_npcs):
                dialogue = await npc_conversation(npc, nearby_npc)
                await ws_broadcast(session, {
                    "type": "npc_dialogue",
                    "npcs": [npc_id, nearby_npc.id],
                    "content": dialogue
                })
        
        await asyncio.sleep(SIM_STEP_INTERVAL)  # 仿真步间隔
```

### 4.4 NPC-NPC 对话系统

```
触发条件（两个NPC在同一场景+都在"可对话"状态）：

场景：戏台
  陈师傅 (在拉琴) + 小华 (在练功)
    ↓
  后台检测到 proximity
    ↓
  LLM 生成短对话（3-5轮，~200 tokens）
    ↓
  对话记录存入双方记忆流
    ↓
  前端通过 WebSocket 显示气泡："陈师傅：手腕再松一点。"
```

**玩家看到的体验**：
- 走过戏台时，看到陈师傅和小华的气泡交替出现
- 可以"偷听"（停下来看完整对话）
- 也可以直接走过去打断（按F键进入玩家对话，NPC-NPC对话中断）

### 4.5 与章节系统的协作

这是最关键的设计。NPC自主行为**不能**破坏章节推进：

| 章节约束 | NPC 行为限制 |
|---------|-------------|
| 第一章需要与小华对话 | 小华不会离开戏台超过30分钟 |
| 第三章需要陈师傅揭露真相 | 陈师傅的 autonomy_level 降低，"空闲时间"增加（方便玩家找到他） |
| 第五章需要所有NPC支持 | 所有NPC保持"可对话"状态，不会进入"不在服务区" |

**实现**：叙事导演（系统四）向每个NPC下发`autonomy_level`（0=完全手动 到 5=完全自主），关键章节节点降低自主等级。

## 5. 系统二：记忆网络系统

### 5.1 设计目标

将梨园生死现有的"对话历史列表"升级为三维记忆网络，使NPC能够：
1. 语义检索相关记忆（而非全量注入）
2. 知道空间信息（"茶馆里有什么"）
3. 形成关系记忆（"那个玩家帮过我"）

### 5.2 三层记忆架构

```
┌─────────────────────────────────────────────────┐
│              记忆流 (Memory Stream)               │
│  按时间排序的完整事件记录（最底层，不检索）          │
│  每10分钟自动总结一次（重要性归零触发反思）         │
├─────────────────────────────────────────────────┤
│          关联记忆 (Associative Memory)             │
│  语义检索：Recency + Relevance + Importance 加权    │
│  用于：对话时注入相关上下文                        │
├─────────────────────────────────────────────────┤
│          空间记忆 (Spatial Memory)                 │
│  MemoryTree: 梨溪镇 → 茶馆 → 柜台 → 京胡           │
│  用于：NPC知道自己在哪、周围有什么                  │
└─────────────────────────────────────────────────┘
```

### 5.3 关联记忆：对话升级

**改造前**（梨园生死现有）：
```python
# prompt_builder.py — 注入全部对话历史
dialogue_history = session.npcs[npc_id].dialogue_history[-10:]  # 最近10轮
system_content += format_history(dialogue_history)
```
**问题**：10轮对话可能2000 tokens，但只有2轮是相关的。

**改造后**（融合版）：
```python
# 斯坦福小镇式检索
relevant_memories = associative_memory.retrieve(
    current_context=npc.current_situation,  # "玩家在问我父亲的过去"
    k=5,  # 只取最相关的5条
    weighting={
        "recency": 0.3,     # 越近越重要
        "relevance": 0.5,   # LLM评分的语义相关度
        "importance": 0.2   # 人工标注的重要事件
    }
)
# 5条记忆 ≈ 300 tokens vs 10轮对话 ≈ 2000 tokens
```

### 5.4 空间记忆：NPC知道自己在哪

```yaml
# spatial_memory.yaml — 每个NPC独立构建

# 陈师傅的空间记忆（示例）：
npc_chen:
  root:
    - sector: 茶馆
      arenas:
        - arena: 一楼大厅
          objects:
            - "柜台（梅姨常在这里招呼客人）"
            - "靠窗的桌子（我每天在这拉琴）"
            - "墙上的京胡（父亲的遗物）"
        - arena: 二楼
          objects:
            - "我的房间"
            - "旧木箱（藏着当年的戏服）"
    - sector: 戏台
      arenas:
        - arena: 舞台
          objects:
            - "破旧的地板（第三块会吱呀响）"
            - "幕布（三十年没换了）"
        - arena: 后台
          objects:
            - "化妆台"
            - "小华的铺盖"
```

**使用场景**：
- 玩家说"带我去看看你拉琴的地方" → NPC检索空间记忆 → "在茶馆，靠窗那张桌子"
- NPC日程说"去戏台" → 用空间记忆+路径规划找到去戏台的路
- 玩家展示物品"这把京胡你认识吗" → NPC检索空间记忆 → "当然认识！挂在茶馆墙上那把，你父亲留下的"

### 5.5 关系记忆：超越数值

**现有系统**：`npc.relationship = 55`（一个数字）

**融合后**：
```python
{
    "npc_chen": {
        "relationship_value": 55,  # 保留数值（向后兼容）
        "memory_density": 0.73,     # 记忆密度 = 有效记忆数/总互动次数
        "key_shared_experiences": [
            {"event": "玩家帮修屋顶", "importance": 8, "emotion": "感激"},
            {"event": "玩家问了父亲的戏", "importance": 6, "emotion": "悲伤"},
            {"event": "玩家展示了京胡", "importance": 9, "emotion": "震惊+怀念"}
        ],
        "knowledge_about_player": [
            "是柳三秋的儿子（但他自己还不知道）",
            "对戏曲有天生的直觉（身体记忆）",
            "真心关心戏班（不是来猎奇的）"
        ]
    }
}
```

**玩家体验的提升**：
- 不是看到"陈师傅好感度 +5"，而是明天再来时听到陈师傅说"昨天那把京胡……我翻来覆去看了半夜"
- NPC对你的态度不再是数字驱动的if-else，而是基于共享记忆的连续体验
- 第一章帮小华的忙→第五章小华还记得→不是因为"好感度到100了"，而是因为那条记忆被检索到了

## 6. 系统三：世界活态感知系统

### 6.1 设计目标

让梨溪镇从6张静态PNG变成一个"会呼吸"的世界。

### 6.2 时间流逝系统

```
游戏内时间系统：

游戏中 1 天 = 现实 30 分钟（可调）

  06:00 日出 → 色调暖黄
  09:00 上午 → NPC开始活动
  12:00 正午 → 路人最多
  15:00 下午 → 
  18:00 黄昏 → 色调橘红
  20:00 夜晚 → 色调暗蓝，灯笼亮起
  22:00 深夜 → 大部分NPC休息

时间流逝速度可调整：
  T = 1x（正常速度，适合日常探索）
  T = 3x（加速，玩家在等待下一个章节节点时）
  T = 0x（暂停，玩家在对话或查看物品时自动暂停）
```

**前端实现**：
- 色调渐变（使用现有tintOverlay机制）
- NPC按照日报表移动/消失/出现
- 场景物品按时间变化（早上的茶馆热气腾腾，晚上的茶馆烛光摇曳）

### 6.3 天气系统

```
天气枚举：晴 → 多云 → 小雨 → 大雨 → 晴（循环）

天气影响：
  - 大雨时NPC倾向室内活动
  - 小雨时NPC可能"在码头看雨"
  - 天气是NPC对话的话题源："这雨下得……当年你父亲走的那天，也是这样的雨。"
```

## 7. 系统四：叙事导演系统

### 7.1 设计目标

这是梨园生死独有的系统——斯坦福小镇没有"叙事控制"概念，而这正是梨园生死的灵魂。

**叙事导演的职责**：在NPC自主行为和章节叙事之间建立平衡。它不控制NPC说什么，但控制"什么时候什么NPC可以说什么层次的话"。

### 7.2 三层控制粒度

```
层次一：节奏控制（每章）
  - 定义本章的 autonomy_level（0-5）
  - 定义本章的关键事件（触发章节完成）

层次二：阶段控制（每章内的3个阶段）
  - stage_1: 探索期（NPC自由度高，玩家自由探索）
  - stage_2: 发展期（NPC开始主动接近玩家，抛出线索）
  - stage_3: 高潮期（NPC约束高，集中在关键场景）

层次三：时刻控制（关键节点）
  - 当玩家触发"发现孩童戏服"事件 → 导演提升陈师傅的autonomy
  - 当玩家在码头站了5分钟 → 导演让老李主动走过来搭话
```

### 7.3 具体剧本：以第一章为例

```
第一章《闻声·异样》叙事导演编排：

舞台期 (前30%探索)
  └─ NPC行为：正常日程（小华在戏台练功，老周在后台打盹）
  └─ 玩家自由探索小镇
  └─ 导演监控：玩家是否接近戏台？

发展期 (30%-60%线索收集)
  └─ 玩家进入戏台 → 导演触发"身体共鸣"事件
  └─ 小华注意到玩家：“你怎么了？”
  └─ 导演提示老周从后台走出来打圆场
  └─ 关键物品"旧木箱"出现在戏台后台

高潮期 (60%-100%真相揭示)
  └─ 玩家与小华对话≥3轮 → 小华透露"陈师傅说这里有个天才"
  └─ 玩家找到旧木箱 → 打开发现孩童戏服
  └─ 导演确认：玩家已接触到"戏班神童"线索
  └─ 小华 + 老周投票 → 章节完成
```

### 7.4 与 NPC 自主系统的协作

```
叙事导演 ⇄ NPC自主系统 协作协议：

1. 导演通过 autonomy_level 控制NPC自由程度
2. 导演通过 constrained_areas 控制NPC活动范围
3. 导演通过 idle_dialogue_topics 控制NPC闲聊话题
4. NPC自主行为产生的"有趣事件"可以被导演捕获并升级为"关键事件"
5. 导演可以"快进"时间（玩家等了太久没进展，加速日程推进）

合作示例：
  - NPC老李自主决定"去茶馆喝茶"
  - 闲聊时和梅姨说起"今天码头来了个陌生人"
  - 导演检测到"陌生人来访"这个事件 → 评估是否可以升级为线索
  - 如果符合当前章节需求 → 导演让梅姨在下次见玩家时主动提起
```


## 10. 新的游戏循环设计

### 10.1 三层循环

```
时刻层 (0-30秒)
  └─ 行动：WASD移动 / F对话 / 拾取物品
  └─ 反馈：NPC气泡 / 对话SSE流式 / 物品旁白
  └─ 奖励：关系值变化 / 记忆碎片解锁 / 章节进度

叙事层 (5-30分钟)
  └─ 目标：完成当前子任务 / 发现关键物品 / 与关键NPC对话
  └─ 张力：时间流逝 + NPC自主行为（可能错过某些对话）
  └─ 解决：子任务完成 → NPC投票 → 章节推进

记忆层 (小时-多周)
  └─ 进展：章节完成 / NPC关系网络演变
  └─ 回响：之前章节的选择影响后续NPC态度（通过记忆检索，非脚本）
  └─ 结局：基于整场游戏的记忆密度和关键选择
```

### 10.2 玩家一天的游戏体验（设计稿）

```
06:00 游戏内时间 → 玩家从茶馆二楼醒来
                  → 梅姨已经在准备早茶
                  → "早啊，昨儿你睡得真沉。"
                  → [玩家可以选择帮梅姨备茶 / 直接出门]

08:00 → 走到戏台
      → 看到小华在练功，陈师傅在拉琴（NPC-NPC对话进行中）
      → 气泡："手腕放松！""我知道啦——"
      → [玩家可以偷听 / 打断对话]

10:00 → 在码头发现老李在修船
      → 对话中老李提到"昨晚有个人在戏台那边烧纸"
      → 这是由"时间流逝+天气"触发的自主事件（不是脚本）

14:00 → 回到茶馆，梅姨说："今早你不在，陈师傅来找过你。"
      → 这是NPC自主行为的结果（陈师傅按日程来茶馆送新茶）
      → 玩家选择去戏台找陈师傅 → 触发关键对话

18:00 → 黄昏，色调变暖
      → 所有NPC走回各自场所
      → 玩家可以总结今天收集的线索
```

## 11. 风险评估与缓解

### 11.1 技术风险

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| LLM调用频率暴增 | 高 | NPC自主行为可能导致LLM调用量×5 | `importance_filter`阈值+行为缓存（相同日程缓存1h） |
| WebSocket不稳定 | 中 | NPC位置不同步，出现"瞬移" | HTTP轮询fallback，每5秒同步位置 |
| 记忆检索延迟 | 中 | 对话前检索耗时过长 | 本地向量索引（sqlite-vss）替代LLM打分 |
| 叙事一致性破坏 | 高 | NPC自主行为与章节冲突 | 导演autonomy_level动态限制（章节高潮期降为0） |

### 11.2 设计风险

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| "太像斯坦福小镇" | 低 | 失去梨园特色 | 保持章节驱动的核心不变，只增强NPC智能 |
| NPC太自主导致玩家失控 | 中 | 玩家不知道该做什么 | 叙事导演 + TaskPanel引导（现有系统已很强） |
| 复杂度爆炸 | 高 | 调试困难 | 分4阶段实施，每阶段可验证可回滚 |

## 附录：关键设计决策记录

### A. 为什么不用斯坦福小镇的"完全自由NPC"？

梨园生死是**叙事游戏**，不是**模拟器**。玩家来听一个故事，不是来观察蚂蚁窝。完全自由的NPC会导致：
- 关键对话NPC可能"不在服务区"
- 故事节奏完全失控
- 玩家不知道该找谁、该做什么

所以采用"约束式自主"——NPC有生活，但在章节框架内。

### B. 为什么保留关系值数值？

玩家需要**进展感**。纯记忆密度模型虽然真实，但玩家感受不到"我做了什么让NPC更喜欢我"。数值+记忆并行：数值给玩家即时反馈（"陈师傅态度转变了"），记忆给NPC行为深度（"因为他记得你帮他修过屋顶"）。

### C. 为什么不用斯坦福小镇的Django+Phaser.js技术栈？

梨园生死的FastAPI+Vite+Phaser 3已经成熟。斯坦福小镇提供的是**认知架构**，不是技术栈。移植的是算法和设计模式，不是框架。
