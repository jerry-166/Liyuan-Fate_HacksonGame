## 2. 未来愿景与产品定位

### 2.1 产品愿景

> **梨园生死不是「一款游戏」，而是一个「AI叙事游戏引擎 + 首个旗舰剧本」。**
>
> 我们卖的第一件商品是《梨园生死》——一个完整的、2-3小时的江南戏曲叙事体验。但产品真正的价值在于：同一套引擎可以支撑无限个剧本。从江南小镇到赛博朋克，从古代武侠到现代都市——换个剧本文件就是一款新游戏。

### 2.2 目标用户画像

| 用户类型 | 描述 | 市场规模 | 核心诉求 |
|---------|------|---------|---------|
| **叙事游戏爱好者** | 喜欢《去月球》《底特律变人》类体验 | 中等 | 情感共鸣、独特故事 |
| **AI技术尝鲜者** | 对AI驱动叙事好奇 | 大 | "每次玩都不一样"的体验 |
| **戏曲/国风爱好者** | 对传统文化有兴趣 | 中等但付费强 | 文化沉浸、美术表现力 |
| **独立游戏开发者** | 想用我们引擎做自己的剧本 | 小但商业价值高 | 剧本编辑器、发布平台 |
| **Vtuber/主播** | 需要"有节目效果"的内容 | 大（传播价值） | 不可预测的剧情走向 |

### 2.3 核心竞争力

不是「AI对话」，不是「AI叙事」——这些竞品也能做。我们的护城河是**四合一**：

```
① NPC认知架构（斯坦福小镇赋能）
   ×
② 章节式动态叙事引擎（梨园原创）
   ×
③ 可插拔剧本框架（已设计）
   ×
④ 完整商业化管线（安全/存档/多语言/多平台）
```

竞品能做好其中1-2个，但四个全部打通的公司——目前没有。

### 2.4 产品形态规划

```
梨园生死 v1.0（黑客松版）
  └─ 单一剧本、技术验证

梨园生死 v2.0（9月11交付版）
  ├─ 旗舰剧本《梨园生死》  ← 完整商业化体验
  ├─ 内置剧本《XX》× 2     ← 展示框架能力
  └─ 剧本开发工具包          ← 吸引开发者

梨园生死 Platform（2027+）
  ├─ 剧本商店
  ├─ 玩家创作社区
  └─ AI辅助剧本生成
```

## 3. 重大架构决策

### 3.1 决策一：游戏引擎 → **保持 Phaser 3 + 增强，不切换**

| 选项 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| Phaser 3（当前） | 2D成熟、Web原生、团队熟悉、部署简单 | 3D能力弱、性能有上限 | ✅ **保持** |
| Godot 4.x | 免费开源、2D/3D兼顾、GDscript | 团队需学习、Web导出不如Phaser | ❌ |
| Unity | 生态最大、3D强大 | 太重、WebGL体积大、许可证费 | ❌ |
| Cocos Creator | 国产、小游戏适配好 | 社区小、AI绑定弱 | ❌ |

**理由**：
1. 梨园生死是**2D叙事游戏**，不是3D动作游戏。Phaser 3的2D能力完全足够
2. 现有代码资产（tilemap、sprite、collision、scene management）全部可复用
3. Web原生 → 天然支持Web/Steam（Electron）/移动端（PWA或Capacitor）
4. 团队3人已有Phaser经验，切换引擎 = 白费2-4周生产力
5. **核心创新在AI架构，不在渲染层**。用最好的AI + 够用的引擎 = 最优解

**需要增强的部分**：
- Phaser → v3.80+（最新稳定版，修复了内存泄漏）
- 引入 Spine 骨骼动画（NPC表情/动作）
- 引入 WebGL 后处理（色调渐变、天气粒子、景深）
- 性能Profile（地图140×100瓦片，当前80×50，需验证）

### 3.2 决策二：后端架构 → **FastAPI + asyncio + WebSocket，不切换**

| 选项 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| FastAPI（当前） | 异步原生、SSE/WS支持好、Pydantic验证 | 生态不如Django | ✅ **保持** |
| Django | 生态全、admin后台 | 异步支持不原生、重 | ❌ |
| Go/Fiber | 性能极致 | Python AI生态无法复用 | ❌ |
| Node.js/Express | 前端同语言 | AI调用不如Python | ❌ |

**理由**：AI生态100%绑定Python。FastAPI的async/await + SSE流式 + WebSocket是这3个月最需要的能力。

**需要增强的部分**：
- WebSocket服务：NPC实时位置推送、NPC-NPC对话气泡
- 后台仿真循环：`asyncio.create_task()` 运行NPC自主行为
- 写分离：SQLite读 → Redis缓存；SQLite写 → asyncio线程池

### 3.3 决策三：数据库 → **SQLite（不变）+ Redis缓存层**

| 选项 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| SQLite（当前） | 零配置、嵌入式、WAL模式 | 写入串行、高并发瓶颈 | ✅ **保持 + Redis** |
| PostgreSQL | 并发强、全文搜索 | 运维成本、单机不需要 | ❌ |
| MongoDB | 文档灵活 | AI游戏不需要 | ❌ |

**理由**：
1. 梨园生死是**单机/小规模游戏**，不是MMO。SQLite WAL模式的并发读取完全够用
2. 存档系统已建立在SQLite上，迁移成本>收益
3. Redis缓存层解决高频读取（NPC状态、记忆检索结果、LLM响应的短期缓存）
4. 如果未来上Steam → SQLite嵌入客户端，零运维

**优化路径**：
```
当前: 每轮对话全量UPSERT 5个NPC → 1ms（可接受）
v2.0: NPC增至15个 → 改为只UPSERT变更NPC（P1方案已设计）
v3.0: 多人在线 → PostgreSQL迁移（届时再评估）
```

### 3.4 决策四：LLM架构 → **分层调用 + 模型路由器**

当前是单一模型调用。商业化需要**成本可控 + 质量保证**：

```
LLM路由器设计：

┌──────────────────────────────────────────────┐
│              LLM Router                       │
│                                               │
│  请求分析 → 重要性判定 → 模型选择              │
│                                               │
│  IMPORTANCE HIGH (15%)                        │
│    → GPT-4o / Claude-3.5-Sonnet               │
│    → 场景：章节任务规划、结局生成、关键对话      │
│                                               │
│  IMPORTANCE MEDIUM (35%)                       │
│    → GPT-4o-mini / Qwen-Plus                   │
│    → 场景：日常对话、物品发现旁白、NPC投票       │
│                                               │
│  IMPORTANCE LOW (50%)                          │
│    → 本地模型 / 缓存 / 模板                    │
│    → 场景：NPC间闲聊、问候语、天气预报          │
└──────────────────────────────────────────────┘
```

**成本估算**（一局游戏，约2小时）：
- 全用GPT-4o：约$8-12
- 使用分层路由：约$1.5-3
- 成本降低：70-80%

### 3.5 决策五：部署方案 → **CloudStudio MVP → 独立云服务器 → Steam**

| 阶段 | 方案 | 时间 |
|------|------|------|
| 黑客松 | CloudStudio静态部署 | ✅ 已完成 |
| 9月11交付 | 腾讯云轻量服务器 / 阿里云ECS | 6月-7月 |
| Steam发行 | Electron封装 + NW.js或Tauri | 8月-9月 |
| 移动端 | PWA或Capacitor封装 | 9月+ |

### 3.6 决策六：多语言架构

```
// Tauri 封装：Steam
├── zh-CN/（原生）
├── en/（AI辅助翻译 + 人工校对）
└── ja/（优先级低于英语）

翻译策略：
- UI文本：人力翻译（量小，~500条）
- 剧本/人设：AI翻译 + 人工校对
- NPC实时对话：AI生成时指定目标语言（不是翻译，是直接生成）
```

## 4. 斯坦福小镇深度融合方案

### 4.1 融合原则

> **梨园生死的叙事结构是骨架，斯坦福小镇的认知架构是血肉。不替换，只增强。**

梨园生死已有的系统（章节引擎、物品系统、子场景、存档、安全、BGM）**全部保留**。斯坦福小镇提供的是**NPC认知能力**——这是梨园最缺的东西。

### 4.2 四大融合系统

详见 `深度融合方案_v2.md`，此处聚焦**与商业化目标的对应关系**：

| 融合系统 | 商业化价值 | 优先级 |
|---------|-----------|--------|
| **NPC自主生活系统** | 让小镇"活"起来 → 沉浸感↑ → 好评↑ | P0 |
| **记忆网络系统** | NPC"记得"玩家 → 情感连接↑ → 复玩↑ | P0 |
| **世界活态感知系统** | 日夜循环+天气 → Steam截图传播↑ | P1 |
| **叙事导演系统** | 节奏控制 → 不卡关不暴走 → 完播率↑ | P0 |

### 4.3 可立即开始的最小可行增强（2周）

这是从斯坦福小镇移植过来的**最轻量但效果最显著**的部分：

```
Week 1-2 目标：让陈师傅"活起来"

1. 关联记忆检索（移植 AssociativeMemory）
   - 替换 prompt_builder 的全量对话历史注入
   - 效果：Token消耗 -50%，NPC能回顾更早的对话

2. 空间记忆（移植 MemoryTree，适配梨溪镇）
   - 为6个子场景构建空间语义
   - 效果：NPC知道"京胡挂在茶馆墙上"

3. NPC基础日程（改造 Plan/Execute 模块）
   - 陈师傅每日：茶馆→戏台→茶馆→茶馆二楼
   - 效果：陈师傅会在地图上移动
```

## 10. 附录：技术决策备忘录

### A. 为什么不做3D？

梨园生死的核心创新在**AI叙事**，不在**视觉技术**。2D让你：
- 更快迭代（美术资源周期短）
- 更广的设备覆盖（低配PC也能跑）
- 更低成本（2D原画师薪资 < 3D建模师）
- 更聚焦（玩家注意力在对话和情感，不在多边形）

3D是2027年的规划，不是2026年的。

### B. 为什么不用斯坦福小镇的Django？

已分析过。FastAPI的async/await + WebSocket + SSE比Django更适合实时AI游戏。斯坦福小镇提供的是**认知架构**，不是技术栈。

### C. 为什么保持章节系统？

完全自由的NPC（像斯坦福小镇那样）在没有章节的框架下是"观察蚁穴"，有章节框架才是"体验故事"。梨园的核心是「可控涌现」——章节是骨架，NPC自主是血肉。去掉骨架，血肉散落一地。

### D. 为什么保留关系值数值？

虽然斯坦福小镇证明非数值化的记忆密度可以做社会模拟，但：
- 游戏玩家需要 **进展感**（"我做了什么让NPC更亲近我"）
- 数值给即时反馈（关系值变化 +5 / -3）
- 记忆给行为深度（"他记得你帮他修过屋顶"）
- 两者不是排斥的，是互补的

### E. 为什么9月11日？

3个月是游戏从MVP到商业化的最小可行周期。更早 = 质量不够；更晚 = 团队burnout + 市场窗口风险。
