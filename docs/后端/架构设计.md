## 二、技术选型

### 2.1 FastAPI

| 对比维度 | FastAPI ✅ | Flask | Django |
|---------|-----------|-------|--------|
| 异步支持 | 原生 async/await | 需扩展 | 需 ASGI 适配 |
| SSE 流式 | 原生 StreamingResponse | 需手动处理 | 需 Channels |
| 数据校验 | Pydantic 内置 | 需手动 | DRF Serializer |
| WebSocket | 原生支持 | 需 flask-sock | 需 Channels |
| 性能 | 接近 Node.js | 一般 | 一般 |

### 2.2 自实现 Agent 框架

| 对比维度 | 自实现 ✅ | LangChain | CrewAI |
|---------|----------|-----------|--------|
| 流式控制 | 完全可控 | 受限于框架抽象 | 基于 LangChain |
| 学习成本 | 极低 | 高 | 中 |
| 依赖体积 | 零额外依赖 | ~200MB+ | ~150MB+ |
| MVP 适配度 | ★★★★★ | ★★ | ★ |

**选择理由**：MVP 只有 2 个 NPC Agent，SSE 逐 token 输出需要对 LLM 响应流的精确控制，自实现核心代码量 < 300 行。

### 2.3 LLM 调用（httpx + OpenAI 兼容接口）

- `httpx` 原生支持 `async` + 流式响应
- OpenAI 兼容接口是腾讯云 LLM 的标准对接方式
- `aiter_lines()` 逐行解析 SSE，完美控制每个 delta token

> **容错策略**：`tenacity` 指数退避重试（最多 2 次）；`asyncio.Semaphore` 控制并发上限；模型通过 `LLM_MODEL` 环境变量可配置。

### 2.4 状态管理（内存 + SQLite）

```
内存层（热数据 - O(1) 读写）         持久层（冷数据 - SQLite）
┌──────────────────────────┐      ┌──────────────────────┐
│ sessions: dict[str,      │      │ dialogues 表          │
│           GameSession]   │◄────►│ events 表             │
│                          │      │ sessions 表           │
│ GameSession              │      └──────────────────────┘
│ ├── session_id           │
│ ├── player_name          │        恢复流程：session_id 存前端 localStorage
│ ├── api_key (内存仅存)    │        → 刷新时从 SQLite 重建 → 载入内存 dict
│ ├── current_stage        │
│ ├── npcs: dict           │
│ └── events_triggered     │
└──────────────────────────┘
```

**Session 生命周期**：Lazy Loading（请求时重建）→ TTL 淘汰（2 小时不活跃 → 从内存移除，SQLite 保留数据）→ GET /api/health 监控活跃会话数。

## 四、NPC Agent 系统

### 4.1 架构

```
                        ┌─────────────────────┐
                        │    prompt_builder    │
                        │                     │
                        │  拼接以下内容：       │
     ┌──────────────────┤  ① 世界观 (system)   ├──────────────────┐
     │                  │  ② 当前阶段+对话基调  │                  │
     │                  │  ③ NPC 人设卡        │                  │
     ▼                  │  ④ 跨NPC事件注入      │          ┌───────▼──────┐
┌─────────┐             │  ⑤ 对话历史(最近N轮)  │          │   personas/  │
│ 全局状态 │             │  ⑥ 玩家输入          │          │              │
│ stage   │             │  ⑦ 对话生成指令       │          │ chen_shifu  │
│ events  │             └──────────┬───────────┘          │   .yaml     │
│ relation│                        │                      │ xiaohua     │
└─────────┘                        ▼                      │   .yaml     │
                          ┌─────────────────┐             └──────────────┘
                          │    LLM Client    │
                          │  流式调用 → token │
                          └────────┬────────┘
                                   ▼
                          ┌─────────────────┐
                          │ response_parser │
                          │ 提取:           │
                          │ • dialogue_text │
                          │ • options[]     │
                          │ • events[]      │
                          │ • relation_delta│
                          └────────┬────────┘
                                   ▼
                          ┌────────▼────────┐
                          │ 阶段判定 LLM    │ ← 独立轻量调用
                          │ (非嵌入对话prompt)│
                          └────────┬────────┘
                                   ▼
                           SSE → 前端
```

> **设计决策**：阶段判定从对话 prompt 中拆出为独立 LLM 调用，避免小模型对复杂 prompt 的多任务遵从度不稳定，以及对话生成与阶段判定耦合导致的连锁崩溃。

**Response Parser 降级策略**：
- `options` 解析失败 → 空选项列表
- `events` 解析失败 → 跳过事件触发
- `relationship_delta` 解析失败 → 默认 0（配合规则兜底）
- `dialogue_text` 提取失败 → 使用 LLM 原始输出全文

### 4.2 NPC 人设卡格式（YAML）

```yaml
# personas/chen_shifu.yaml
id: npc_chen

  1: # 不屑阶段
    attitude: "冷淡疏离，话中带刺"
```

### 4.3 NPC Handoff 机制

```
玩家与 NPC A 对话 → 触发事件 → 事件写入 session.events_triggered
                                    │
玩家与 NPC B 对话 → prompt_builder 检测到事件 → 注入跨NPC上下文
                                    │
                              NPC B 识别事件 → 回应自然衔接
```

**MVP 事件定义**：

| 事件 ID | 触发 NPC | 触发条件 | 影响 |
|---------|---------|---------|------|
| `chen_sent_to_xiaohua` | npc_chen | 陈师傅让玩家找小华 | 小华识别跨NPC上下文 |
| `xiaohua_mentioned_chen` | npc_xiaohua | 小华提到陈师傅往事 | 陈师傅识别跨NPC上下文 |
| `chen_told_past` | npc_chen | 陈师傅讲起戏班三十年前 | 阶段推进候选 |

### 4.4 对话退出与轮数限制

三层退出机制：
1. **NPC 自然判断结束**：System Prompt 中加入对话结束规则
2. **对话轮数限制**：默认 `MAX_DIALOGUE_ROUNDS=10`，达到上限时注入强结束指令
3. **显式退出接口** `POST /api/dialogue/exit`

> 📎 **关系值系统详见 [_shared/关系值系统.md](../_shared/关系值系统.md)** — 包含 Clamp 机制、事件奖励表、实现代码。

## 六、对话摘要（Summary）

### MVP 策略：延后实现

MVP 阶段对话轮数预计 < 20 轮，上下文窗口（通常 4K-8K tokens）足够容纳完整历史。后续版本方案：

```
对话轮次超过阈值（如 15 轮）→ 触发摘要（前10轮 → 300字摘要）
→ 替换上下文（摘要 + 最近5轮原文）→ 标记 summarized=true
```
