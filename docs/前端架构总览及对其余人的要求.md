## 二、架构模式与目录结构

### 2.1 架构模式：**Phaser Scene 场景驱动架构**

项目使用 Phaser 3 内置的 **Scene（场景）生命周期系统**，本质上是：

| 层级 | 对应概念 | 说明 |
|------|---------|------|
| **Scene 层** | 类似"视图层" | 每个 Scene 管理一组完整的游戏画面和逻辑 |
| **Event Emitter 层** | 类似"消息总线" | Scene 间通过 `scene.events` 进行解耦通信 |
| **API 层** | 数据访问层 | `src/api/client.js` 封装所有后端通信 |
| **Config 层** | 常量配置 | `src/config.js` 存放游戏常量、配色、场景 key |

这不是经典的 MVC 或组件化，而是 Phaser 原生的**场景驱动 + 全局事件总线**模式——每个 Scene 独立管理自己的渲染循环（`create` + `update`），通过事件系统横向通信。

### 2.2 目录结构与划分依据

```
frontend/
├── index.html             # 入口 HTML，挂载 #game-container
├── package.json           # 依赖：仅 phaser + vite
├── vite.config.js         # Vite 配置（proxy /api 到 8000）
└── src/
    ├── main.js            # 游戏启动入口，Phaser.Game 实例化
    ├── config.js          # 全局常量（尺寸、颜色、场景 key、阶段色调）
    ├── api/
    │   └── client.js      # API 客户端（Mock 数据 + 真实 HTTP 调用）
    └── scenes/
        ├── BootScene.js   # 启动场景（空加载，直接转 GameScene）
        ├── GameScene.js   # 核心游戏场景（地图、玩家、NPC、交互）
        └── UIScene.js     # UI 覆盖层场景（TODO：对话 UI / HUD）
```

**划分依据**：完全基于 **Phaser Scene 生命周期**：

- **BootScene**：初始化/资源加载（当前是空壳，不做任何加载）
- **GameScene**：游戏世界本身——地图渲染、玩家控制、NPC 放置、碰撞检测、交互检测
- **UIScene**：运行在 GameScene 之上的 UI 覆盖层，用 `this.scene.launch('UIScene')` 并行启动，独立管理渲染深度
- **config.js** 独立是因为多个 Scene 共享同一套常量
- **api/client.js** 独立是为了将网络通信与游戏渲染完全解耦

## 三、核心组件分析

### 3.1 `main.js` — 游戏入口

**职责**：创建 `Phaser.Game` 实例，配置引擎参数。

```16:25:frontend/src/main.js
const config = {
  type: Phaser.AUTO,
  parent: 'game-container',
  width: 1024,
  height: 768,
  backgroundColor: '#1a1a2e',
  pixelArt: true,
  roundPixels: true,
  scale: {
    mode: Phaser.Scale.FIT,
    autoCenter: Phaser.Scale.CENTER_BOTH,
  },
  physics: {
    default: 'arcade',
    arcade: {
      gravity: { x: 0, y: 0 },
      debug: false,
    },
  },
  scene: [BootScene, GameScene, UIScene],
};
```

**关键设计决策**：
- **1024×768** 固定分辨率，`Phaser.Scale.FIT` 自动适配屏幕
- **Arcade Physics** 仅用于碰撞检测（`gravity: 0` 无重力）
- **`pixelArt: true` + `roundPixels: true`** → 像素风渲染，无抗锯齿模糊
- **Scene 数组顺序**：`BootScene` → `GameScene` → `UIScene`，按默认注册顺序依次启动

### 3.3 `GameScene.js` — 核心游戏场景 ⭐

这是项目最大的文件（640 行），承载了 MVP 当前几乎全部的游戏逻辑。以下是内部模块拆解：

#### 3.3.1 地图系统（程序化生成，非 Tiled）

**设计选择**：**程序化生成**而非加载外部 tilemap 文件。80×50 格子，每格 16px = 1280×800 世界。

- 12 种 Tile 类型：草地、青石板路、土路、河水、石桥、戏台、茶馆、码头、祠堂、民居、树木
- 每种 Tile 有 3 色调色板：`base` / `dark` / `highlight`，配合棋盘格（`(r+c)%2===1`）实现视觉变化
- 程序化布局：主街、小河、桥、戏台区域（4×14）、茶馆、码头、祠堂、散落民居
- 每种 Tile 有独立的**纹理细节绘制逻辑**（水波纹、石板缝隙、戏台飞檐、茶馆茶旗等）

## 四、全局状态管理方案

### 4.1 当前状态管理分析

目前前端**没有独立的全局状态管理**（无 Redux / Zustand / Pinia）：

| 状态类别 | 位置 | 管理方式 |
|---------|------|---------|
| 游戏配置常量 | `config.js` | 静态导出常量 |
| 玩家位置/速度 | `GameScene.player` | Phaser Sprite 实例属性 |
| NPC 信息 | `GameScene.npcs[]` | Sprite 实例 + `setData('key', value)` |
| 当前交互 NPC | `GameScene.currentNearbyNPC` | 场景实例属性，每帧更新 |
| 游戏状态（阶段/关系值等） | `api/client.js` **Mock** | 静态 JSON |
| 会话 ID | `localStorage` | 由前端持久化，后端 API 返回 |

**要点**：几乎所有**业务状态**（阶段、关系值、事件、对话历史）目前**仅在 Mock 数据中**，真实数据将由后端管理。前端只负责"发起请求 → 渲染响应"。

### 4.2 数据流向

```
[用户操作]
    │
    ▼
GameScene.update()              ← Phaser 游戏循环（60fps）
    │
    ├── WASD → player.setVelocity()
    ├── overlap 检测 → 显示 [F] 提示
    └── JustDown(F) → emit('dialogue:start')
                           │
                           ▼
                    UIScene (监听事件)
                           │
                           ▼
                    api/client.js
                    ├── startGame()          → POST /api/game/start → 初始化游戏
                    ├── getGameState()       → GET /api/game/{id}  → 拉取状态
                    ├── startDialogueStream() → POST /api/dialogue → SSE 流式对话
                    └── evaluateEnding()     → POST /api/game/{id}/evaluate → 结局评价
                           │
                           ▼
                    后端 FastAPI (B 开发中)
```

**关键设计**：当前 `USE_MOCK = true`，所有 API 调用走 Mock 数据；`setUseMock(false)` 后切换到真实后端。

### 4.3 SSE 流式处理架构

`client.js` 中的 `startDialogueStream()` 设计：

```89:103:frontend/src/api/client.js
export async function startDialogueStream(sessionId, npcId, playerMessage = null) {
  if (USE_MOCK) {
    return createMockSSEStream();
  }
  const res = await fetch(`${BASE}/dialogue`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ session_id: sessionId, npc_id: npcId, player_message: playerMessage })
  });
  if (!res.ok) throw new Error(await res.text());
  return res.body;
}
```

**设计特点**：
- 选用 `fetch` + `ReadableStream` 而非 `EventSource`——因为 `POST` 请求才能发送 `player_message`
- Mock 模式用 `createMockSSEStream()` 模拟真实流式效果（加了 300ms 延迟）
- 真实模式直接返回 `res.body`（`ReadableStream`），由调用方逐 chunk 解析 SSE 事件

**Mock SSE 事件结构**（与 API 文档完全对齐）：

```
event: delta
data: {"chunk": "戏班啊……"}

event: delta
data: {"chunk": "三十年前，这镇上的戏台可是夜夜满座。"}

event: done
data: {"full_text":"...", "options": [...], "stage_changed": false, "new_stage": null, ...}
```

## 五、路由配置与页面映射

### 5.1 路由方案

**没有传统路由**。Phaser 3 是一个单页面游戏引擎，所有"页面切换"通过 **Scene 切换** 实现：

| 逻辑"页面" | Phaser Scene | 触发方式 |
|------------|-------------|---------|
| 启动/加载 | `BootScene` | 游戏初始化时自动启动 |
| 游戏主世界 | `GameScene` | `this.scene.start('GameScene')` |
| 对话界面 | `UIScene`（覆盖层） | `this.scene.launch('UIScene')` |

**目前仅为 2 层结构**：`GameScene`（底层地图+角色） + `UIScene`（上层 UI）。未来扩展的"子场景入口"（进入茶馆内部、进入祠堂内部）可能通过新的 Scene 或 GameScene 内部的"模式切换"实现。

| **7** | 音箱/音乐：`STAGE_TONES` 预留了 `bgm_mood` 字段但未实现 | C 的 AI 音乐产出需要与阶段色调一一对应，且播放接口需接入 Phaser Audio | 🟢 P2 |

### 综合协作建议

```
当前最大缺口布局：

前端 (人A)          后端 (人B)            内容 (人C)
──────────────────────────────────────────────────
✅ 地图渲染          🔴 后端代码未开始    🔴 NPC人设卡未交付
✅ WASD控制          🟡 API 已设计        🔴 tile/角色图未交付
🔴 UIScene空白       ✅ Mock 数据已提供    🟡 微剧本大纲已讨论
🟡 对话API封装完成   🟡 架构设计方案完成   🟡 音乐/音效未启动
```

**MVP 关键路径**：`A 实现 UIScene(对话UI)` ← 依赖 → `B 完成后端核心链路`、`C 提供 NPC 人设 + 基础美术`。三者需要**同步推进**，不能用"等某个人完成再开始"的水瀑布模式。

## 九、总结与改进建议

### 优势
1. **架构清晰**：Scene 驱动 + API 层解耦，职责划分干净
2. **Mock 先行**：前端可以在不依赖后端的情况下独立开发
3. **API 契约对齐**：`client.js` 的 4 个函数与 API 文档 4 个接口完全对应
4. **程序化占位巧妙**：在无美术资产时用地道的手绘 Graphics 呈现，足够 MVP 验证

### 风险
1. **🔴 UIScene 空白是最大风险**：对话 UI 是 MVP 核心交互，却尚未开始
2. **🔴 无 SSE 解析逻辑**：`startDialogueStream` 返回了 `ReadableStream`，但没有任何代码消费它
3. **🟡 状态管理缺失**：缺少对 `stage_changed`、`relationship_change`、`events_triggered` 的事件响应逻辑
4. **🟡 无错误处理**：`client.js` 的 try-catch 和 `error` SSE 事件均未处理
5. **🟢 无场景切换**：子场景（茶馆内部、戏台后台）切换逻辑尚未设计
