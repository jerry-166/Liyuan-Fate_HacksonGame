## 二、技术选型

### 2.1 后端框架：FastAPI

| 对比维度 | FastAPI ✅ | Flask | Django |
|---------|-----------|-------|--------|
| 异步支持 | 原生 async/await | 需扩展 | 需 ASGI 适配 |
| SSE 流式 | 原生 StreamingResponse | 需手动处理 | 需 Channels |
| 数据校验 | Pydantic 内置 | 需手动 | DRF Serializer |
| WebSocket | 原生支持 | 需 flask-sock | 需 Channels |
| 性能 | 接近 Node.js | 一般 | 一般 |

**选择理由**：SSE 流式对话是核心链路，async + Pydantic 内置校验完美匹配 MVP 需求。

### 2.2 Agent 框架：自实现（纯 Python）

| 对比维度 | 自实现 ✅ | LangChain | CrewAI |
|---------|----------|-----------|--------|
| 流式控制 | 完全可控 | 受限于框架抽象 | 基于 LangChain |
| 学习成本 | 极低 | 高（Chain/Agent/Tool 概念） | 中 |
| 依赖体积 | 零额外依赖 | ~200MB+ | ~150MB+ |
| 调试难度 | 直接 print/log | 链路不透明 | 链路不透明 |
| MVP 适配度 | ★★★★★ | ★★ | ★ |

**选择理由**：
- MVP 只有 2 个 NPC Agent，不需要 Multi-Agent 编排框架
- SSE 逐 token 输出需要对 LLM 响应流的精确控制
- 自实现 Agent 核心代码量 < 300 行

**Agent 核心逻辑**：
```python
class NPCAgent:
    def __init__(self, persona: Persona, session: GameSession):
        self.persona = persona
        self.session = session

    async def generate_response(self, player_message: str) -> AsyncIterator[str]:
        prompt = await self._build_prompt(player_message)
        async for chunk in llm_client.stream_chat(prompt):
            yield chunk  # 逐 token 产出，直通 SSE
```

### 2.3 LLM 调用：httpx + OpenAI 兼容接口

```
腾讯云 LLM (混元/DeepSeek) → OpenAI 兼容 HTTP API → httpx 异步调用 → SSE 流式解析
```

**选择理由**：
- `httpx` 原生支持 `async` + 流式响应
- OpenAI 兼容接口是腾讯云 LLM 的标准对接方式
- `aiter_lines()` 逐行解析 SSE，完美控制每个 delta token

**LLM Client 核心逻辑**：

> **容错策略**：
> - `tenacity` 指数退避重试（最多 2 次），应对网络抖动 / API 限流
> - `asyncio.Semaphore` 控制并发 LLM 调用数，防止 QPS 超限
> - 模型通过环境变量 `LLM_MODEL` 可配置，早期用 `hunyuan-lite` 验证链路，随时可切更强模型

```python
from tenacity import retry, stop_after_attempt, wait_exponential

class LLMClient:
    def __init__(self, api_key: str, base_url: str = "https://api.hunyuan.cloud.tencent.com/v1",
                 model: str = "hunyuan-lite", max_concurrency: int = 5):
        self.client = httpx.AsyncClient(timeout=30.0)
        self.api_key = api_key
        self.base_url = base_url
        self.model = model
        self._semaphore = asyncio.Semaphore(max_concurrency)

    @retry(stop=stop_after_attempt(2), wait=wait_exponential(min=1, max=5))
    async def stream_chat(self, messages: list[dict]) -> AsyncIterator[str]:
        async with self._semaphore:
            async with self.client.stream(
                "POST", f"{self.base_url}/chat/completions",
                headers={"Authorization": f"Bearer {self.api_key}"},
                json={"model": self.model, "messages": messages, "stream": True}
            ) as response:
                async for line in response.aiter_lines():
                    if line.startswith("data: ") and line != "data: [DONE]":
                        chunk = json.loads(line[6:])
                        if delta := chunk["choices"][0]["delta"].get("content"):
                            yield delta
```

### 2.4 状态管理：Session 级状态机

```
内存层（热数据 - O(1) 读写）         持久层（冷数据 - SQLite）
┌──────────────────────────┐      ┌──────────────────────┐
│ sessions: dict[str,      │      │ dialogues 表          │
│           GameSession]   │◄────►│ events 表             │
│                          │      │ sessions 表           │
│ GameSession              │      └──────────────────────┘
│ ├── session_id           │
│ ├── player_name          │        恢复流程：
│ ├── api_key (内存仅存)    │        session_id 存前端 localStorage
│ ├── current_stage        │        → 刷新时从 SQLite 重建 GameSession
│ ├── npcs: dict           │        → 热数据载入内存 dict
│ ├── events_triggered     │
│ ├── game_ended           │
│ └── ending_type          │
└──────────────────────────┘
```

**选择理由**：
- 内存 dict：对话请求需要即时读写 NPC 关系和阶段状态，O(1) 是必须的
- SQLite：对话历史和事件日志不需要实时查询，SQLite 足够且零运维
- 恢复机制：前端 localStorage 存 session_id，刷新后可重建完整状态

**Session 生命周期管理**：

```
启动时：
  → 不主动加载所有 session（避免内存浪费）
  → 启动时内存 dict 为空

请求时（lazy loading）：
  → session_id 查内存 dict → 命中 → 直接使用
  → 未命中 → 从 SQLite 重建 GameSession → 载入内存 dict

TTL 淘汰（后台清理）：
  → 会话超过 2 小时未活跃 → 从内存移除（SQLite 保留完整数据）
  → 前端恢复时触发 lazy loading 即可重建

GET /api/health 端点：
  → 返回 {"active_sessions": N, "uptime": ...}，方便调试监控
```

## 四、NPC Agent 系统

### 4.1 Agent 架构

```
                        ┌─────────────────────┐
                        │    prompt_builder    │
                        │                     │
                        │  拼接以下内容：       │
     ┌──────────────────┤  ① 世界观 (system)   ├──────────────────┐
     │                  │  ② 当前阶段+对话基调  │                  │
     │                  │  ③ NPC 人设卡        │                  │
     ▼                  │  ④ 跨NPC事件注入      │          ┌───────▼──────┐
┌─────────┐             │  ⑤ 对话历史(最近N轮)  │          │   personas/  │
│ 全局状态 │             │  ⑥ 玩家输入          │          │              │
│ stage   │             │  ⑦ 对话生成指令       │          │ chen_shifu  │
│ events  │             └──────────┬───────────┘          │   .yaml     │
│ relation│                        │                      │ xiaohua     │
└─────────┘                        ▼                      │   .yaml     │
                          ┌─────────────────┐             └──────────────┘
                          │    LLM Client    │
                          │  流式调用 → token │
                          └────────┬────────┘
                                   ▼
                          ┌─────────────────┐
                          │ response_parser │
                          │ 提取:           │
                          │ • dialogue_text │
                          │ • options[]     │
                          │ • events[]      │
                          │ • relation_delta│
                          │                 │
                          │ ⚠️ 降级策略：    │
                          │ 解析失败 →      │
                          │  options=[]     │
                          │  events=[]      │
                          │  delta=0        │
                          └────────┬────────┘
                                   │
                          ┌────────▼────────┐
                          │ 阶段判定 LLM    │ ← 独立轻量调用
                          │ (非嵌入对话prompt)│
                          │ 输入：摘要+状态  │
                          │ 输出：是否推进   │
                          └────────┬────────┘
                                   ▼
                           SSE → 前端
```

> **设计决策：阶段判定从对话 prompt 中拆出**
>
> 早期设计将阶段判定指令嵌入对话 prompt 末尾，让 LLM 在生成对话的同时判定阶段切换。但这存在两个问题：
> 1. 小模型（hunyuan-lite）对复杂 prompt 的多任务遵从度不稳定
> 2. 对话生成与阶段判定耦合，一处解析失败会连锁崩溃
>
> 改为独立调用后，对话 prompt 更短更聚焦，阶段判定 prompt 也可独立优化，互不影响。

> **response_parser 降级策略**
>
> LLM 输出格式不可控时，必须保证对话链路不断：
> - `options` 解析失败 → 返回空选项列表，前端不显示选项按钮
> - `events` 解析失败 → 跳过事件触发，不影响对话继续
> - `relationship_delta` 解析失败 → 默认 0（配合 §4.4 的规则兜底增量）
> - `dialogue_text` 提取失败 → 使用 LLM 原始输出全文作为回复

### 4.3 NPC Handoff 机制

```
玩家与 NPC A 对话 → 触发事件 → 事件写入 session.events_triggered
                                    │
玩家与 NPC B 对话 → prompt_builder 检测到事件 → 注入跨NPC上下文
                                    │
                              NPC B 识别事件 → 回应自然衔接
```

**流程示例**：

```
1. 玩家对陈师傅说"我想学戏，但不知道该从哪开始"
   → 陈师傅回复："去找小华，他欠我个人情，让他教你基本功"
   → LLM 响应触发事件: "chen_sent_player_to_xiaohua"

2. 玩家走到小华处，按 F 对话
   → prompt_builder 检测: "chen_sent_player_to_xiaohua" ∈ events
   → 注入 prompt: "[跨NPC事件] 陈师傅让这位玩家来找你学基本功"
   → 小华回复: "陈师傅让你来找我的？啧，他居然还记着这事……行吧，你先扎个马步我看看"
```

**事件定义（MVP）**：

| 事件 ID | 触发 NPC | 触发条件 | 影响 |
|---------|---------|---------|------|
| `chen_sent_to_xiaohua` | npc_chen | 陈师傅让玩家找小华 | 小华识别跨NPC上下文 |
| `xiaohua_mentioned_chen` | npc_xiaohua | 小华提到陈师傅往事 | 陈师傅识别跨NPC上下文 |
| `chen_told_past` | npc_chen | 陈师傅讲起戏班三十年前 | 阶段推进候选 |

### 4.5 对话退出与轮数限制 (v1.1 新增)

**三层退出机制**：

1. **NPC 自然判断结束**：System Prompt 中加入"对话结束规则"，LLM 在话题穷尽时自然收尾，options 为空数组。
2. **对话轮数限制**：单次与同一 NPC 的连续对话上限为 `MAX_DIALOGUE_ROUNDS`（默认 10）。达到上限时 prompt 自动注入强结束指令；接近上限（8+轮）时注入渐进收尾提示。
3. **显式退出接口** `POST /api/dialogue/exit`：前端点 ESC 或退出按钮时调用，让 NPC 生成告别语，并标记 `is_available=false`。

```python
# config.py
MAX_DIALOGUE_ROUNDS = 10

# NPCState 新增字段
dialogue_round_count: int = 0
```

### 4.4 关系值 Clamp 机制

> **问题**：若完全交给 LLM 判断 `relationship_delta`，小模型可能输出 +50 或 -50 等极端值，导致关系值失控、阶段意外切换。

**策略：LLM 建议 + 规则兜底 + Clamp 限制**

```
每轮对话结束时：

1. LLM 返回 relationship_delta（建议值）
   ↓
2. Clamp 限制：将 delta 限制到 [-5, +10] 范围
   - 正向上限 +10：防止一轮对话关系暴涨
   - 负向下限 -5：防止一轮对话关系暴跌
   ↓
3. 规则兜底增量（LLM 解析失败时）：
   - 每轮对话固定 +3（NPC 不可能越聊越讨厌你）
   - 触发关键事件额外 +5~+10（由事件定义表指定）
   ↓
4. 累加到 NPC.relationship，最终 clamp 到 [-100, 100]
```

**实现**：

```python
# 规则兜底增量表
EVENT_RELATIONSHIP_BONUS = {
    "chen_sent_to_xiaohua": 5,
    "xiaohua_mentioned_chen": 5,
    "chen_told_past": 10,
}

def apply_relationship_delta(npc: NPCState, llm_delta: int | None,
                              events: list[str]) -> int:
    """计算并应用关系值变化，返回实际变化量"""
    if llm_delta is not None:
        # LLM 建议 → clamp 到安全范围
        delta = max(-5, min(10, llm_delta))
    else:
        # 规则兜底：每轮固定 +3
        delta = 3

    # 事件额外加分
    for event_id in events:
        delta += EVENT_RELATIONSHIP_BONUS.get(event_id, 0)

    # 累加并 clamp 到总范围
    old = npc.relationship
    npc.relationship = max(-100, min(100, npc.relationship + delta))
    return npc.relationship - old  # 实际变化量
```

## 五、阶段切换引擎

### 5.1 加权阶段判定架构

> **设计决策**：阶段判定从对话 prompt 中拆出，作为独立轻量 LLM 调用。采用加权策略：规则达标立即切换；仅 LLM 判定则需连续 2 轮一致才切换。

```
每轮对话完成后（SSE done 事件生成时）：

   ┌──────────────────┐
   │  规则引擎 O(1)    │
   │                  │
   │ 条件1: any NPC   │
   │  relationship   │
   │  >= 50 → true   │
   │                  │
   │ 条件2: 关键事件   │
   │  已触发 → true   │
   └────────┬─────────┘
            │
     ┌──────┴───────┐
     │ 规则命中？     │
     └──┬──────┬────┘
        │      │
     Yes│      │No
        ▼      ▼
   立即切换  ┌──────────────────┐
            │ 独立 LLM 阶段判定  │
            │                  │
            │ Prompt（轻量）：   │
            │ "当前阶段{stage}， │
            │  关系值{rel}，    │
            │  已触发事件{evts}，│
            │  最近对话摘要。    │
            │  是否可推进？"     │
            │                  │
            │ 输出：            │
            │ {should_advance: │
            │  bool, reason}   │
            └────────┬─────────┘
                     │
              ┌──────┴───────┐
              │ LLM 判定推进？ │
              └──┬──────┬────┘
                 │      │
              Yes│      │No
                 ▼      ▼
        ┌────────────┐  跳过
        │ 连续判定计数 │
        │             │
        │ 连续 2 轮   │──── No ──→ 不切换，计数清零
        │ 都说推进？   │
        └──────┬──────┘
               │ Yes
               ▼
          执行阶段切换
```

**加权规则总结**：

| 判定来源 | 权重 | 切换条件 |
|---------|------|---------|
| 规则引擎 | **立即生效** | 任一条件达标 → 直接切换，无需 LLM 确认 |
| LLM 判定 | **需连续确认** | 连续 2 轮 LLM 独立判定均为"可推进" → 才执行切换 |
| 两者同时 | 规则优先 | 规则命中时不等 LLM 结果 |

**选择理由**：
- 规则引擎是确定性护栏，不存在幻觉问题，命中即切换
- LLM 可能因叙事上下文误判，连续 2 轮确认可有效过滤幻觉
- 独立调用使对话 prompt 更短更聚焦，阶段判定 prompt 也可独立优化

### 5.2 阶段参数

| 阶段 | 名称 | 色调 | BGM | 对话基调 | 切换条件 |
|------|------|------|-----|---------|---------|
| 1 | 不屑 | cold | melancholy | 冷漠疏离、话中带刺 | 游戏开始 |
| 2 | 了解 | warm | hopeful | 温和敞开、偶有真情 | 关系达标 OR 关键事件 OR LLM 判定 |
| 3 | 抉择 | dramatic | intense | 情感浓烈、命运抉择 | 关系达标 OR 关键事件 OR LLM 判定 |

### 5.3 阶段切换时的状态变更

```python
async def advance_stage(session: GameSession, new_stage: Stage, reason: str):
    session.current_stage = new_stage
    # 1. 刷新所有 NPC 的问候语（基于新阶段+关系值）
    for npc in session.npcs.values():
        npc.current_greeting = await generate_greeting(npc, new_stage)
    # 2. 记录阶段切换事件
    session.events_triggered.add(f"stage_{new_stage.value}_reached")
    # 3. 重置 LLM 连续判定计数（阶段已切换，计数归零）
    session.stage_llm_consecutive = 0
    # 4. 持久化到 SQLite
    await db.update_session(session)
    # 5. 通知前端 → SSE done 事件携带 new_stage

async def check_stage_advancement(session: GameSession, events_this_turn: list[str]):
    """每轮对话完成后调用，判断是否应推进阶段"""
    # 规则引擎：确定性条件，命中立即切换
    if any(npc.relationship >= 50 for npc in session.npcs.values()):
        await advance_stage(session, next_stage(session.current_stage), "规则：关系值达标")
        return True
    if any(e in session.events_triggered for e in KEY_STAGE_EVENTS):
        await advance_stage(session, next_stage(session.current_stage), "规则：关键事件触发")
        return True

    # LLM 独立判定：需连续 2 轮确认
    llm_result = await llm_client.stage_check(
        current_stage=session.current_stage,
        relationships={npc.id: npc.relationship for npc in session.npcs.values()},
        events=session.events_triggered,
        recent_summary=get_recent_summary(session)
    )
    if llm_result.should_advance:
        session.stage_llm_consecutive += 1
        if session.stage_llm_consecutive >= 2:
            await advance_stage(session, next_stage(session.current_stage),
                              f"LLM 连续判定：{llm_result.reason}")
            return True
    else:
        session.stage_llm_consecutive = 0  # 中断则清零

    return False
```

## 六、对话摘要（Summary）

### 6.1 MVP 策略：延后实现，Schema 预留

## 七、API Key 管理

### 7.1 设计原则

- **零存储**：API Key 不写入数据库、日志、文件
- **Session 隔离**：每个游戏会话独立 Key
- **优先级链**：

```
用户前端输入 → 环境变量 TENCENT_LLM_API_KEY → 错误提示
    (最高)              (fallback)                (无Key)
```

### 7.2 实现

```python
# config.py
def resolve_api_key(user_input: str | None = None) -> str:
    """按优先级解析 API Key"""
    if user_input and user_input.strip():
        return user_input.strip()
    env_key = os.environ.get("TENCENT_LLM_API_KEY", "")
    if env_key:
        return env_key
    raise APIKeyMissingError("请提供 LLM API Key")

# routes/game.py
@router.post("/game/start")
async def start_game(request: StartGameRequest):
    try:
        api_key = resolve_api_key(request.api_key)
    except APIKeyMissingError:
        return JSONResponse(
            status_code=400,
            content={"error": True, "code": "API_KEY_MISSING",
                     "message": "请在前端输入您的 LLM API Key"}
        )
    session = state_manager.create_session(
        player_name=request.player_name,
        api_key=api_key  # 仅存内存
    )
```

## 八、SQLite 数据库设计

### 8.1 建表语句

> **WAL 模式**：初始化时执行 `PRAGMA journal_mode=WAL;`，提升高频写入时的并发安全。
> **异步写入**：对话写入用 `asyncio.to_thread()` 包裹，避免阻塞事件循环。

```sql
-- 启用 WAL 模式（应用启动时执行一次）
PRAGMA journal_mode=WAL;


-- sessions 表：会话元数据
CREATE TABLE sessions (
    session_id   TEXT PRIMARY KEY,
    player_name  TEXT NOT NULL DEFAULT '玩家',
    current_stage INTEGER NOT NULL DEFAULT 1,
    game_ended   INTEGER NOT NULL DEFAULT 0,  -- 0/1
    ending_type  TEXT,
    ending_data  TEXT,                         -- JSON: 缓存的结局评价
    deleted      INTEGER NOT NULL DEFAULT 0,   -- 0/1: 软删除
    created_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- npc_states 表：NPC 状态快照
CREATE TABLE npc_states (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id   TEXT NOT NULL,
    npc_id       TEXT NOT NULL,
    relationship INTEGER NOT NULL DEFAULT 0,
    is_available INTEGER NOT NULL DEFAULT 1,
    current_greeting TEXT DEFAULT '',
    FOREIGN KEY (session_id) REFERENCES sessions(session_id)
);

-- dialogues 表：对话历史
CREATE TABLE dialogues (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id   TEXT NOT NULL,
    npc_id       TEXT NOT NULL,
    role         TEXT NOT NULL,  -- 'player' | 'npc'
    content      TEXT NOT NULL,
    options      TEXT,            -- JSON: NPC 回复的对话选项
    stage        INTEGER,
    summarized   INTEGER DEFAULT 0,
    summary      TEXT,
    created_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_dialogue_session ON dialogues(session_id, npc_id);

-- events 表：已触发事件日志
CREATE TABLE events (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id   TEXT NOT NULL,
    event_id     TEXT NOT NULL,
    triggered_by TEXT,           -- npc_id or 'system'
    created_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (session_id) REFERENCES sessions(session_id)

CREATE INDEX idx_events_session ON events(session_id);

-- npc_memories 表：NPC 关键记忆（摘要预留）
CREATE TABLE npc_memories (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id   TEXT NOT NULL,
    npc_id       TEXT NOT NULL,
    content      TEXT NOT NULL,
    importance   INTEGER DEFAULT 1,
    created_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (session_id) REFERENCES sessions(session_id)
);
```
