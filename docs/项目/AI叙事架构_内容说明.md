AI 驱动动态叙事系统架构
========================

五层结构，从上到下逐层驱动，层与层之间通过数据和事件流动。

第一层：开发者输入层 — 骨架

开发者提供三样东西作为游戏的基础骨架（约 200 行 YAML 配置）：

- 微剧本 YAML — 六章结构：序章 + 第一~五章，每章定义标题、描述、触发条件、目标事件、成功条件、子任务模板
- NPC 人设文件 — 5 个核心角色各自的 YAML 文件，包含姓名、年龄、角色定位、性格特征、说话风格、口头禅、背景故事、对话示例、不同阶段的问候语
- 场景与物品定义 — 6 个室内子场景（戏台、茶馆、码头、祠堂、父亲故居、墓地）的地图、碰撞区、NPC 站位，以及 10 件剧情物品的定义

第二层：AI 剧情规划层 — 大脑

游戏运行时，AI 分两个层级规划剧情：

- 开局生成大纲 — 游戏启动时，AI 读取世界观设定和章节骨架，一次性生成五章叙事大纲（每章摘要、核心冲突、氛围）。作为后续逐章规划的参照框架。
- 逐章生成详细任务 — 每进入一章，AI 根据当前游戏状态（已完成章节、NPC 关系值、背包物品、已触发事件）生成该章的 TaskInstance。包含章节描述、子任务列表（每章 3-5 个子任务，链式解锁），关联相关 NPC 和物品。
- 真实因果链 — 前章的行为结果直接影响后章的剧情生成。玩家在第一章的选择和关系变化，会成为第二章任务规划时的上下文输入。不是预写好的分支树，而是 AI 根据实际状态实时生长。

第三层：NPC 智能对话层 — 灵魂

5 个 NPC 全部由 LLM 实时扮演，每轮对话都是完整的上下文注入：

- System Prompt 包含五部分上下文：
  1. 世界观设定（戏班、江南小镇、时代背景）
  2. NPC 角色设定（姓名、性格、说话风格、口头禅、背景故事）
  3. 当前章节信息（章节名称、核心冲突、剧情目标）
  4. 游戏全局状态（所有 NPC 关系值、当前任务进度、玩家背包、已触发事件）
  5. 对话风格示例（预设的玩家-NPC 对话范例）

- User Prompt 包含当轮信息：
  - 玩家输入内容（自由文本或预设选项）
  - 当前关系值
  - 任务进度投票提示（让 NPC 判断相关子任务是否完成）
  - 物品展示上下文（如果玩家正在展示某件物品）
  - 对话轮数限制提示（接近上限时提示自然收尾）

- 输出结构（每轮对话返回）：
  - dialogue_text — NPC 的自然语言回复（2-5 句，SSE 流式输出）
  - relationship_delta — 本轮态度变化（-5 到 +10）
  - options — 3-4 个下一步对话选项（自然结束时为空）
  - should_trigger_event / new_event — 是否触发关键剧情事件
  - task_progress — 该 NPC 对当前任务进度的投票（是否完成、完成哪些子任务）

- 纯文本/JSON 自适应 — LLM 输出自动判断模式。JSON 模式提取 dialogue_text 字段流式展示给玩家，纯文本模式直接流式展示。

第四层：状态管理与章节推进层 — 引擎

游戏状态（GameState）：
- 章节进度 — current_chapter_id、completed_chapters、chapter_outlines（AI 生成的大纲缓存）
- NPC 关系值 — 5 个 NPC 各自的关系值（-100 到 100），每次对话后根据 relationship_delta 实时更新
- 物品背包 — 已发现的 NarrativeItem 列表，每件物品有 base_description、AI 生成的 ai_detail、各 NPC 对该物品的认知（npc_knowledge）
- 已触发事件 — 全局事件集合（如 "chen_tells_past"），触发后记录并用于后续上下文
- 对话全局计数器 — 跨 NPC 统一编号，保证对话历史的时间排序

章节推进引擎 — 双模判定 + NPC 共识投票：

- 规则检测（O(1) 查表，确定性护栏）：
  - 关系值阈值检测 — 任意 NPC 关系值达到当前阶段阈值
  - 关键事件检测 — 触发集合中包含阶段定义的关键事件
  - 最少对话轮数 — 作为附件条件，对话轮数不足时规则不触发

- LLM 独立判定（防规则漏判）：
  - 每次对话后 LLM 独立判断是否应该推进阶段
  - 不计入单次判断，必须连续 2 轮一致才执行切换（防幻觉和随机波动）

- NPC 共识投票（章节级别的推进机制）：
  - 每个相关 NPC 在对话中通过 task_progress 字段投票
  - 双重门槛：LLM 投票通过 + 该 NPC 的对话轮数达到子任务的 min_dialogue_rounds
  - 所有相关 NPC 一致投票通过 → 章节完成 → 自动进入下一章

- 6 种子任务模式：
  - dialogue — 与指定 NPC 对话达到最少轮数
  - acquire_item — 获取指定物品
  - show_item — 向指定 NPC 展示物品
  - deliver — 将物品交付给指定 NPC
  - relation — 与指定 NPC 的关系值达到阈值
  - explore — 到达指定场景并与其中 NPC 对话

- 子任务链式解锁 — 前一个子任务完成后自动解锁下一个

第五层：结局生成层 — 终章

不是 A/B 选择结局，而是整局所有行为累积塑造的专属结局：

- 结局 Header 生成 — AI 回顾全程关键事件和玩家行为，生成：
  - ending_title — 专属结局标题
  - ending_summary — 结局总结
  - key_moments — 整局关键瞬间回顾
  - philosophy — 人生感悟

- 每个 NPC 的个人终章 — AI 根据该 NPC 与玩家的最终关系值、对话历史片段，为每个 NPC 生成一段结局描述（npc_title + npc_ending）。
  - 关系值高 → 温暖的结局
  - 关系值低 → 遗憾或疏离的结局

- 流式渐进展示 — 结局画面通过 SSE 流式逐步显示，玩家依次看到标题、总结、每个 NPC 的结局卡片、最终的人生感悟，形成完整的叙事收束体验。

核心设计理念

开发者只给骨架（微剧本 YAML + 人设 + 场景），AI 在游戏运行中长出血肉（对话、任务、结局），玩家通过自由探索和对话决定故事走向。同一套框架，换个微剧本文件就能演绎完全不同的故事。
