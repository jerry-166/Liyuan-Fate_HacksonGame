## 1. `persist_session()` 每轮对话全量 UPSERT 所有 NPC 状态

### 位置
- `backend/state/manager.py` 的 `persist_session()` 方法
- `backend/storage/database.py` 的 `save_npc_states_batch()` 方法

### 问题描述
核心接口 `POST /api/dialogue`（SSE 流式对话）每一轮都会调用 `persist_session()`，该方法内部通过 `save_npc_states_batch()` 在一个事务中对**所有 NPC** 做全量 UPSERT，即使本轮对话只有一个 NPC 的状态发生了变化。

```python
# manager.py persist_session()
self._db.save_npc_states_batch(session.session_id, session.npcs)
```

### 当前影响
- 当前仅 5 个 NPC，全量 UPSERT 耗时约 0.5~1ms，几乎无影响
- LLM 流式生成耗时 2~8s，完全掩盖了这部分开销

### 恶化条件
- NPC 数量增长到 **10+** 时，全量 UPSERT 耗时会显著增加
- 高并发（多玩家同时在线）场景下会成为瓶颈

### 优化方向（按优先级）

**P1 — NPC 增多时可做：** 改为只 UPSERT 实际变更的 NPC

```python
def persist_session(self, session: GameSession, changed_npc_id: str = None) -> None:
    self._db.update_session(...)
    if changed_npc_id:
        self.persist_npc_state(session, changed_npc_id)
    else:
        self._db.save_npc_states_batch(session.session_id, session.npcs)
```

**P2 — 高并发时：**
- 异步写库：用 `asyncio.to_thread()` 把 SQLite 写操作扔到线程池
- 写合并：同一 session 的多次 persist 合并为一次事务提交

### 风险评估
- 当前阶段（Hackathon / MVP）：**低风险**，无需立即处理
- 正式上线 / NPC 增多后：**中风险**，建议采用 P1 方案
